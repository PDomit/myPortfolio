DOUBLE CLUTCH GEARBOX MECHATRONICS TESTING UNIT

I designed and built a device for testing tightness of the mechatronics of the DSG DQ200 dual-clutch transmission gearbox. The project involved both hardware and software development from scratch. The device is controlled from a PC application created by me and fully controls operation of the mechatronics. The application window is quite simple to use. It has buttons to change valves positions or to start the hydraulic pump. This device helps in detecting faults. A common fault in this mechatronics unit is the loss of pressure too fast because of a crack in the mechatronic's body. The device helps to find out if a mechatronics has a crack or not. This project cosists of several main parts which required different knowledge and skills.

How does it work ?

The principle is to start the hydraulic pump by clicking a button in the application window. The pressure rises. When the pressure reaches the appropriate level the pump stops. The device is connected to mechatronic's pressure sensor and measure the pressure every second. When the pressure falls below the minimum level the pump starts again and the time interval between stopping and starting the pump is shown in the application window. This time interval helps to detect if the pressure drops too quickly. The procedure can be repeated after changing the position of the valves for example when changing gears.

Electrical circuits

The electrical circuits was designed from scratch in order to controll solenoids and the BLDC motor. The objective was to design circuits capable of handling up to 20A current and that could be controlled by STM32 microcontroller. The BLDC motors used in those mechatronics don't have Hall sensors so I had to design circuits which could conduct commutation based on the back-EMF. The phase voltages had to be measured and filtered to finally provide the information about the rotor position. I soldered it together using a universal board.

Firmware

The STM32 board controls BLDC, valves and communicates with PC via UART. The firmware was created in the C language in STM32 Cube IDE environment. The firmware has not been publicly shared. The photo on the left shows voltages of a phases while the motor is running.

Application

The application helps send instructions to the device as well as display the information from the device like time between stopping and starting the pump. The buttons are for changing the valves positions or for starting the pump. The currently selected variant is marked. At the beginning user have to select COM port and click the 'Connect' button. If PC connects with the device the confirmation message will be displayed. The application works on the Windows operating system.

Future prospects

The device works properly but it needs to be upgraded. Currently I am working on the housing, cooling and some safety elements that are intended to prevent burning the device from overheating or burning out.

MEMORY MANAGEMENT SYSTEM

The memory management system is capable of performing all memory operations on a serial access memory M93C46 with the MICROWIRE™ standard. Currently it can be used with the M93C46 or any memory with the same instruction set. It was designed to write, read, erase data to/from the memory chip. All operations are carried out using UART terminal on a PC. To perform an operation user have to enter appropriate two letters code, data if necessary and accept it by clicking 'Enter' button. A memory are suppsoed to be in x8 bit organization. The system is able to cooperate with any serial memory but the instruction set must be adjusted depending on the memory chip used. In the future it will be extended to simplify the process of selecting instruction sets based on used memory.

Hardware-in-the-Loop simulation

HiL simulation was the topic of my master's thesis. The objective was to build HiL testing station for a car with dual clutch automatic gearbox DQ250. The testing station consists of: HiL simulator, a Simulink-based car model with the DQ250 gearbox, STM32 board acting as gearbox controller and a box with a LCD screen and two potentiometers to set brake and gas pedals positions during tests.

HiL simulator

The simulator belongs to AGH The University of Science and Technology in Kraków. The simulator consists of a computer equipped with measurement cards and software that runs simulations based on input signals from the cards and code, which was generated from the plant model of a car.

Plant model

To create and generate C code from the plant model, I used Simulink and its libraries. I also used the car’s technical data to create a model that was as realistic as possible.

Gearbox's controller

The STM32 board was chosen as the control unit for the gearbox. I developed firmware to control the gearbox based on output signals from the measurement cards. These signals include the car’s velocity, currently selected gears (odd and even), and the positions of the gas and brake pedals. Based on these signals, the controller is able to: shift gears (between even and odd), switch the active clutch, and release both clutches. The car velocity and pedal position signals are analog, while the information about the currently selected gears is encoded in four bits, making these signals digital.

Firmware

The firmware was written in C. I created and debugged it using the STM32CubeIDE environment. Several peripherals such as ADC, timers, and GPIOs were used.

Tests

Finally, I conducted several tests, and the results were analyzed in my master’s thesis. I tested whether the car could operate properly, whether changes in the firmware such as faster clutch engagement or extending the upper engine speed limit could improve the car’s dynamics and to what extent. Finally whether the controller was able to detect faults such as low clutch pressure or gear engagement failure.

Results

The chart on the left is one of many results. It shows the car’s speed during the test. The blue line represents the speed after modifying the firmware, while the orange line shows the previous version.

Single Precision Floating Point numbers multiplication

I created an algorithm in assembly in order to calculate a result of a multiplication of two single precision floating point numbers without a floating point unit. The objective was to create the algorithm and test it using Python.

Assembly code

The function was written in assembly for the ARMv7 architecture. For now the function does not support denormalized numbers so the minimal number the function returns is 2^(-126). The code is avaible on my GitHub.

Testing script

In order to test the algorithm I created a script in Python. The script sends numbers to a MCU with the Cortex M4 via UART and the MCU returns a result of multiplication. Then the script checks if the result is correct. Wrong results are counted and printed.
